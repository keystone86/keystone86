# Keystone86 / Aegis — Tooling and Observability Policy

## Purpose

This document defines the expected tooling interface, aggregate verification commands, and observability expectations for this project.

---

## Prerequisites

- `iverilog` and `vvp` (Icarus Verilog) — required for all RTL simulation targets
- `python3` — required for all script targets
- Run all commands from repo root

---

## Aggregate verification commands

These are the commands that must pass before any handoff is considered valid.

### Full baseline (run all three in order)

```bash
make ucode
make rung0-sim
make rung1-regress
make rung2-regress
```

`rung2-regress` is the comprehensive gate — it runs Rung 0 and Rung 1 baseline checks before running Rung 2. Passing `rung2-regress` with no failures means the complete current baseline is green.

### Abbreviated baseline check

If only confirming a single rung has not regressed:

```bash
make ucode && make rung{N}-sim
```

This is acceptable for targeted verification but does not replace the full regression chain before a handoff.

## Generated artifacts

The following files are generated and must not be manually edited:

| File | Generated by | Regenerate with |
|---|---|---|
| `microcode/build/ucode.hex` | `scripts/ucode_build.py` | `make ucode` |
| `microcode/build/dispatch.hex` | `scripts/ucode_build.py` | `make ucode` |
| `microcode/build/ucode.lst` | `scripts/ucode_build.py` | `make ucode` |
| `microcode/build/dispatch.lst` | `scripts/ucode_build.py` | `make ucode` |

The RTL package and include files (`rtl/include/keystone86_pkg.sv`, `rtl/include/*.svh`) are intended to be generated by `make codegen` but the generator (`tools/spec_codegen/gen_from_appendix_a.py`) is currently a bootstrap stub. Do not edit these files manually — see `docs/implementation/coding_rules/source_of_truth.md`.

## RTL simulation source lists

The exact source files compiled for each rung are defined in the Makefile under `IVERILOG_SOURCES`, `IVERILOG_SOURCES_RUNG1`, and `IVERILOG_SOURCES_RUNG2`. Any RTL change to a file in these lists requires re-running the affected rung's simulation before claiming the baseline is still passing.

Current source list (all rungs share the same core set):

```
rtl/include/keystone86_pkg.sv
rtl/core/bus_interface.sv
rtl/core/prefetch_queue.sv
rtl/core/decoder.sv
rtl/core/microcode_rom.sv
rtl/core/microsequencer.sv
rtl/core/commit_engine.sv
rtl/core/cpu_top.sv
sim/models/bootstrap_mem.sv      (Rung 0 only)
sim/tb/tb_rung{N}_{name}.sv      (rung-specific testbench)
```

## Observability policy

### What observability is present

`cpu_top.sv` exports the following debug ports, which are connected to all rung testbenches:

| Port | What it observes |
|---|---|
| `dbg_eip` | Architectural EIP (primary correctness check) |
| `dbg_mseq_state` | Microsequencer state (FETCH_DECODE return check) |
| `dbg_upc` | Current micro-PC (dispatch routing check) |
| `dbg_entry_id` | Active dispatch entry ID |
| `dbg_dec_entry_id` | Decoder's current entry classification |
| `dbg_endi_pulse` | ENDI event (instruction completion signal) |
| `dbg_fault_pending` | Fault staged in commit_engine |
| `dbg_fault_class` | Fault class code |
| `dbg_decode_done` | Decoder decode_done signal |
| `dbg_fetch_addr` | Current fetch address (deadlock diagnosis) |

These are permanent observability ports. Do not remove them. They are the mechanism by which testbenches prove correctness without waveform inspection.

### What observability is not present

Verbose cycle-by-cycle trace logging is not currently active. If detailed pipeline tracing is needed during development, add it inside `synthesis translate_off` / `synthesis translate_on` guards and remove it before handing off.

Do not leave temporary instrumentation in committed RTL. See `docs/process/post_rung2_cleanup_plan.md` item 8 for the classification criteria.

### Testbench timeout behavior

All three testbenches have timeout watchdogs. On timeout, they print the current values of key debug signals before calling `$finish`. This output is the primary deadlock diagnostic. If a simulation hangs rather than reporting pass/fail, the timeout dump is the starting point for investigation.

## Adding a new rung simulation target

When adding Rung N to the Makefile:

1. Define `IVERILOG_SOURCES_RUNG{N}` with the correct source list.
2. Add `rung{N}-sim`, `rung{N}-regress`, and `rung{N}-clean` targets following the existing pattern.
3. Update `make clean` to include `rung{N}-clean`.
4. Add all three targets to the `help` output.
5. The `rung{N}-regress` target must run `rung{N-1}-regress` (or equivalent) before running the new testbench.
